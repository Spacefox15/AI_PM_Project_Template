Here's how every file fits into your workflow, in the order you'd actually use them:

---

## Phase 1: Define (Before You Write a Single Line of Code)

### `docs/PRD.md` — Start Here, Always
This is your **product requirements document** and the first thing you fill out on any new project. It forces you to answer: *What problem am I solving, for who, and how will I know if it worked?*

In practice: Open this before touching any other file. Write the problem statement, define your success metrics, and list what's explicitly out of scope. If you can't fill this out clearly, you're not ready to build yet. For an AI PM portfolio, a well-written PRD alone is impressive — it shows you think before you build.

---

## Phase 2: Plan (Architecture and Decisions)

### `docs/architecture.md` — Draw Before You Code
Once you know *what* you're building from the PRD, this file helps you figure out *how* it fits together. It has an ASCII system diagram, a component breakdown, and a list of environment variables you'll need.

In practice: Sketch the data flow here — what does the user input, what happens to it, what comes back out? You'll reference this constantly while building and update it when the design changes. Even a rough diagram here saves hours of confusion later.

### `docs/decisions.md` — Your Running Decision Journal
Every time you face a fork in the road (which AI model? which framework? what to defer to v2?), you log it here with your reasoning.

In practice: You won't fill this all at once. You add an entry every time you make a meaningful choice. For example: "I chose vanilla JS over React because I wanted to ship this weekend without a build pipeline." That one sentence, documented, is worth more to a hiring manager than the code itself. It shows PM judgment.

---

## Phase 3: Build (The Actual Coding)

### `.env.example` — Set Up Your Secrets Safely
This is a template showing what environment variables your project needs (like your Anthropic API key) without exposing the actual values.

In practice: When you start a project, copy this file to `.env`, fill in your real API key, and never touch `.env.example` again. The `.gitignore` (below) makes sure your real `.env` never gets pushed to GitHub accidentally.

### `.gitignore` — Protect Yourself From Embarrassing Mistakes
This file tells Git which files to ignore and never commit — things like your `.env` file with your API key, `node_modules/`, and OS junk like `.DS_Store`.

In practice: This file is already written for you. You mostly just leave it alone. But understand what it does — if your API key ever gets pushed to a public GitHub repo, you need to rotate it immediately. This file prevents that.

### `src/index.html` — Your App Shell
The bare-bones HTML structure of your app — the input form, button, output area, and footer.

In practice: This is where you build out your UI. Replace the placeholder labels and text with your actual project content. If you're vibecoding, this is what you point Antigravity at and say "make this look like a real product."

### `src/style.css` — Your Base Visual Design
Clean, professional base styles using CSS custom properties (variables) so you can re-theme each project by changing about 8 color values at the top.

In practice: For each new project, change `--primary` to a new color, swap the font if you want variety, and customize from there. You already know CSS so this will feel natural. The priority badge styles (high/medium/low) are already wired up to match the JSON output structure in `app.js`.

### `src/app.js` — The AI Engine
This is the core logic file — it contains your system prompt, the Anthropic API call, the response parser, and the UI rendering function.

In practice: The three things you customize here per project are: (1) the `SYSTEM_PROMPT` constant — this is your prompt engineering work, and it's where most of your iteration happens; (2) the `renderResults()` function — shaped to match whatever JSON structure your prompt returns; and (3) the event listeners at the bottom — wired to whatever buttons and inputs your HTML has.

---

## Phase 4: Iterate (Track What Changes and Why)

### `CHANGELOG.md` — Your Iteration History
A running log of every version of your project, what was added, changed, or fixed, and links back to decisions you made.

In practice: Every time you make a meaningful update, add an entry at the top. Even if you're the only developer, this habit signals professionalism. It also lets you tell a story in interviews: "Here's v0.1 where I had inconsistent JSON output, here's what I changed in the prompt to fix it by v0.3."

### `.github/pull_request_template.md` — PM Discipline on Every Change
When you open a pull request on GitHub, this template auto-populates with checkboxes and prompts asking you to explain what changed, why, and how you tested it.

In practice: Even solo, get in the habit of making small PRs instead of committing directly to `main`. This creates a readable history of your thinking. If a future employer looks at your GitHub, they'll see someone who ships with intention, not someone who just pushes code and hopes for the best.

---

## Phase 5: Ship and Showcase

### `README.md` — Your Public Product Brief
This is the face of your project. Written like a product brief, not a developer README. It has the problem statement, who it's for, how it works, a live demo link, a results section, and your roadmap.

In practice: You'll actually write most of this *last*, once you know what you built and what it actually does well. The results section is especially important — even if this is a portfolio project, document what you observed in testing. "Processed 150 rows in under 8 seconds, output was correctly structured on 19 of 20 test prompts" is a real result worth showing.
